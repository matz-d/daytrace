# Skill Miner 提案品質に関する問題提起

## 概要

`skill-miner` の E2E 検証において、提案フェーズが「本当に自動化候補として意味のあるパターン」を見つけたというより、圧縮結果から無理に提案件数を満たしているように見える事象が確認された。

今回の検証では、`prepare -> proposal -> selection -> detail -> draft` の流れ自体は完走した。一方で、提案の中身を精査すると、候補の一部は反復パターンとしての根拠が弱く、分類やドラフト化の説得力が不足していた。

この問題は「skill-miner が動かない」問題ではなく、「skill-miner が出す提案をどこまで信頼できるか」という品質上の問題である。ハッカソン審査では特に、自律性とクオリティの両面に影響する。

## 関連参照パス

今回の問題は、以下の実装・仕様・検証導線を参照しながら確認した。

- 問題提起メモ: `ISSUE-skill-miner-proposal-quality.md`

- skill-miner の実行仕様: `plugins/daytrace/skills/skill-miner/SKILL.md`

- detail 再取得 CLI: `plugins/daytrace/scripts/skill_miner_detail.py`
- skill-miner の主要テスト: `plugins/daytrace/scripts/tests/test_skill_miner.py`

今回の E2E では、上記の `prepare -> proposal -> selection -> detail -> draft` フローを通し、その過程で「提案数は出ているが、候補の強さや分類の妥当性が不足している」点を問題として認識した。

また、この検証過程で会話上に生成したスキル作成ドラフトは、現時点では別ファイルとして保存されておらず、**issue*.md に紐づく文脈で作成されたドラフトスキルである**ことも合わせて明記しておく。問題把握の際は、『提案品質の問題を観察するために一度ドラフト化まで進めたが、そのドラフト自体は正式保存していない』という前提で読む必要がある。

---

## 発生した問題

### 1. 提案件数を満たすことが優先され、候補の強さが十分に吟味されていない

実行時には 3-5 件程度の提案を返す前提で進行したが、実データ上は「強い候補が複数ある」とは言いづらかった。

その結果、

- 十分な反復根拠がある候補
- 単発に近い unclustered 候補
- まだ意味解釈が粗い巨大クラスタ

が同じ「提案候補」として並びやすくなっていた。

### 2. 巨大クラスタが意味的に粗すぎる

検証時の出力では、75 packets 中 63 packets が 1 つの candidate cluster に集約された。

このクラスタは、

- `review_changes`
- `search_code`
- `summarize_findings`

のような汎用的な task shape と、

- `rg`
- `sed`
- `read`
- `bash`

のような広く使われる tool signature を共有していたため、大量のセッションが 1 つに寄った可能性が高い。

しかし、この状態では「これは具体的に何を自動化すべき候補なのか」を 1 本に定義しにくい。実際には、

- コードレビュー
- 調査タスク
- ログ整理
- ドキュメント確認

のような近いが別目的の作業が混在している可能性がある。

### 3. unclustered 候補が“単発の面白い話題”として提案されてしまう

unclustered は本来、「再利用可能な反復パターンではないが観測されたセッション」として扱うべき可能性がある。

それにもかかわらず、提案件数を満たすために unclustered から候補を拾うと、

- 反復性の根拠が弱い
- 他セッションへの一般化が弱い
- 一度きりの作業や雑談を誤って skill / hook / CLAUDE.md 候補に見せてしまう

という問題が起きる。

### 4. 5 分類の割り当てが“診断”ではなく“後付けのラベル貼り”になりうる

現状の流れでは、候補が十分に強いかどうかより先に、

- `skill`
- `plugin`
- `agent`
- `CLAUDE.md`
- `hook`

のいずれかに当てはめる圧力がある。

このため、本来は

- まだ分類不能
- 反復はあるが粒度が粗い
- 追加調査が必要
- 候補不成立

と判断すべきケースでも、いずれかの分類に押し込んでしまうリスクがある。

---

## なぜ問題なのか

### 自律性の観点

`skill-miner` は「自分で調べて、価値のある自動化候補を提案する」ことが期待される。しかし、弱い根拠の候補まで提案してしまうと、

- 提案数を埋めることが目的化する
- 候補の妥当性判断を人間がやり直す必要がある
- “自走して価値ある候補を見つけた”というより、“候補らしいものを並べた”状態になる

ため、自律性の評価を下げる。

### クオリティの観点

提案品質が不安定だと、出力体験として

- 「それっぽいが使えない」
- 「なるほどとは思うが、なぜこれが候補なのか弱い」
- 「結局、人が判断し直さないといけない」

となりやすい。これは体験品質の低下に直結する。

### インパクトの観点

skill-miner の魅力は「自分の AI 活用の反復を見つけ、再利用可能な知識に変えること」にある。ここで誤検出が多いと、

- 本当に意味のある反復を見逃す
- 価値の低い候補が混ざる
- 提案そのものへの信頼が落ちる

ため、他者に広がる価値も弱くなる。

---

## 現時点で考えられる原因

## 原因 1. 圧縮フェーズが“広く集める”ことに強く、“意味的に分け切る”ことには限界がある

`skill_miner_prepare.py` は、全セッションを raw history から読み、軽量な特徴量に圧縮して候補を返す構造になっている。

この方式はコンテキスト節約には非常に有効だが、逆に以下の限界がある。

- 代表スニペットが短く、細かな目的差分が消える
- `task_shape` が汎用語彙に寄ると、異なるタスクが同じ block に乗る
- `tool_signature` も `rg`, `sed`, `read`, `bash` のような共通ツールに偏りやすい

つまり、圧縮時点では「似ているが別物」を十分に分離できないケースがある。

## 原因 2. block / similarity の特徴量が汎化しすぎている

現状の類似度は、以下のような要素を重み付きで見ている。

- task shape
- representative snippets
- tool signature
- artifact hints
- repeated rules

これは設計として妥当だが、観測データが開発系タスクに偏ると、

- 多くのセッションが似た tool を使う
- 似た指示語を含む
- 似た出力作法を共有する

ため、類似度が過大評価される可能性がある。

特に `review_changes` / `search_code` / `summarize_findings` のような語彙は、かなり広範な開発作業に出現しうる。

## 原因 3. 「候補が弱い場合は少なく返す / 返さない」という出口設計が弱い

現状の SKILL フローでは、提案フェーズで 3-5 件程度を返す期待が強く見える。

このため、

- 候補が 1 件しか強くない
- どの候補も信頼度が低い
- 追加調査しないと分類できない

といった状況でも、何かしら候補を出してしまいやすい。

## 原因 4. “深掘り調査” が通常フローに組み込まれていない

今回のように巨大クラスタや曖昧候補が出たとき、本来はその場でさらに

- クラスタの内部を再分割する
- 一部の `session_ref` だけ detail を取って粒度を見直す
- 追加の意味解釈を行う

べきだが、現在はそのための明示的なフローが弱い。

その結果、prepare の出力をほぼそのまま提案に変換しやすい。

---

## 今考えうる対応策

## 対応策 A. 提案件数を固定しない

最も重要な対応。

提案フェーズは **0-5 件** を許容し、十分な根拠がある候補だけ返すようにする。

必要なら以下を明示する。

- 強い候補が 0 件なら「今回は有力候補なし」と返す
- 強い候補が 1 件なら 1 件だけ返す
- 弱い候補を件数合わせで補わない

これにより、無理やり提案する圧力を下げられる。

## 対応策 B. 候補ごとに confidence / evidence strength を持たせる

各候補に、少なくとも以下を持たせる。

- `confidence`: `strong` / `medium` / `weak` / `insufficient`
- `evidence_summary`: 何件・何種類のソース・どんな一貫性があるか
- `proposal_ready`: そのままドラフト化できるかどうか

これにより、

- 提案可能
- 追加調査が必要
- 候補不成立

を明確に分けられる。

## 対応策 C. 巨大クラスタ検出時に deep research を発火する

例えば以下の条件で、通常提案を止めて追加調査に移る。

- 1 クラスタが全 packets の 50% 以上を占める
- 共通 tool が `rg`, `sed`, `bash` のような generic なものに偏る
- `representative_examples` の意味がバラバラ
- `near_matches` が多すぎる

このときは、

1. 巨大クラスタだけを対象に再分割を試す
2. `session_refs` を小分けに detail 取得する
3. その結果を基に再分類する

という流れに切り替える。

## 対応策 D. 調査を段階化する

`prepare -> proposal` の間に、必要に応じて中間段階を設ける。

例:

1. `prepare`: 軽量候補生成
2. `triage`: 強い候補 / 曖昧候補 / 候補外 に分ける
3. `research`: 曖昧候補だけ detail 再取得 or 再クラスタ
4. `proposal`: 強い候補だけ正式提案

この段階化により、prepare の出力をそのまま提案に変換しなくて済む。

## 対応策 E. サブエージェント調査を許可する

曖昧な巨大クラスタに対しては、通常の skill フローだけで解釈し切るのではなく、専用の深掘り役を使う方が精度が出る可能性がある。

用途としては、

- クラスタ内部の下位テーマ分解
- 同一パターンと見なす妥当性の検証
- 「本当に skill 化すべきか」の判定

が考えられる。

ただし、毎回サブエージェントに頼るとコストと遅延が増えるため、

- 大型クラスタ
- 信頼度不足
- 候補境界が曖昧

のときだけ発火する段階的エスカレーションが良い。

## 対応策 F. 5 分類の前に「提案不成立」を許可する

現状は 5 分類に入れる前提が強いが、分類前に以下の判定を挟くべき。

- 提案成立
- 追加調査待ち
- 不成立

これにより、分類が目的化するのを防げる。

## 対応策 G. 人間への出力文言を改善する

候補が弱いときは、提案を断定口調で出すのではなく、以下のように明示する。

- 「今回は強い反復候補は 1 件のみです」
- 「以下 2 件は候補未満だが、探索価値があります」
- 「このクラスタは粒度が粗いため、追加調査後の再提案を推奨します」

これだけでも、出力の誠実さと信頼感は大きく改善する。

---

## 優先度付きアクション案

### 優先度高

1. 提案件数を固定せず 0-5 件にする
2. `confidence` / `proposal_ready` を導入する
3. SKILL.md に「弱い候補は無理に提案しない」と明記する
4. 5 分類の前に「追加調査待ち / 不成立」を許可する

### 優先度中

5. 巨大クラスタ検出時の deep research 条件を定義する
6. `triage` 段階を導入する
7. generic tool dominance を再クラスタのトリガーにする

### 優先度低

8. サブエージェント調査を条件付きで導入する
9. 類似度特徴量の見直し（汎用語彙の重み低減、より意味的な差分検出）

---

## この問題提起で目指す状態

最終的に目指すのは、「必ず候補を出す skill-miner」ではなく、以下を正直にできる skill-miner である。

- 強い候補があるときは、自信を持って提案する
- 弱い候補しかないときは、件数を盛らずにそう伝える
- 曖昧なクラスタは追加調査に回す
- 分類不能なら分類不能と判断する

つまり、提案数の多さではなく、**提案の妥当性と説明可能性** を優先する設計に寄せるべきである。

   ---
   name: code-review
   description: >
     差分やファイルを渡すと severity 順の findingsリストを返す。
     指摘ごとにファイル名・行番号を付記し、テスト有無を確認して閉じる。
   user-invocable: true
   ---

   # Code Review

   コード差分またはファイル群をレビューし、構造化された
  findings リストを返す。

   ## Goal

   - 差分・ファイルパスを受け取り、severity 順に指摘を整理する
   - 各指摘にファイル名・行番号の参照を付ける
   - テストカバレッジの有無を確認してからレビューを閉じる

   ## Inputs

   以下のいずれかを受け取る:

   - `git diff` の出力（staged / unstaged / branch 間）
   - 対象ファイルパスのリスト
   - PR 番号（`gh pr diff` で取得）

   ## Output Format

   ### Findings（severity 順）

   | Severity | File:Line | Finding | Suggestion |
   |----------|-----------|---------|------------|
   | 🔴 high | `src/app.py:42` | ... | ... |
   | 🟡 medium | `src/utils.py:18` | ... | ... |
   | 🟢 low | `README.md:5` | ... | ... |

   ### Test Coverage Check

   - [ ] 変更箇所に対応するテストが存在するか確認
   - [ ] テストが不足している場合は追加を提案

   ### Summary

   - 指摘件数: high X / medium Y / low Z
   - 総評: 1-2 文で全体の印象

   ## Execution Rules

   1. まず `git diff` または指定ファイルを読む
   2. `rg` で関連コードを検索し、変更の文脈を把握する
   3. Findings を severity 順にリストアップする
   4. 各 finding にファイル名:行番号を必ず付記する
   5. テストの存在を確認し、不足があれば指摘する
   6. Summary を書いてレビューを閉じる

   ## Rules

   - Findings は必ず severity 降順（high → medium → low）
   - ファイル名・行番号のない指摘は出さない
   - スタイルや好みの指摘は low
  に留め、バグ・セキュリティを優先する
   - テスト確認なしでレビューを閉じない
