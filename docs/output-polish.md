# DayTrace Output Polish v2

## 0. 背景

現状の課題は、日報・提案・投稿下書き・セッション完了サマリといった複数の生成物を、チャット上でそのまま最終成果物として出そうとしていることに起因している。

その結果、以下の問題が出ている。

- テキスト切れ
- 英語や内部状態の混入
- Mixed-Scope 注記やフッター位置の揺れ
- 日報と投稿下書きの役割の混線
- 時系列の矛盾
- 事実と推測の混在
- 選択プロンプトの弱さ
- 「選ばなかった候補」や次回再利用フローの分かりにくさ

このため、チャット出力と最終成果物の責務を分ける方向で再設計する。

---

## 1. 出力アーキテクチャの3層構造化

### Layer 1: Internal Structured Data
役割:
- 正史となる事実データ
- source ごとの scope、timeline、groups、candidate、decision 状態などを保持
- すべての出力の唯一の元データ

要件:
- canonical timeline を持つ
- 事実と推測を分ける
- source 名を正規化する
- 出力用 copy はまだ持たない

### Layer 2: Chat Output (Semi-Final)
役割:
- その場の意思決定のための短い要約
- ユーザーが読むのはここ
- selection prompt や「何が生成されたか」を伝える

要件:
- 長すぎない
- 切れても致命傷にならない
- 最終成果物そのものを無理にここへ載せない
- summary / highlights / action choice に集中する
- ただし **proposal の候補一覧は例外**: 選択判断に必要な最低限の情報（番号 + ラベル + 種類 + 確度 1 行 + アクション導線）は chat に出す（§6 参照）

### Layer 3: Final Markdown Artifact
役割:
- 保存・再利用・共有を前提とした完成版出力
- 日報（自分用/共有用）、投稿下書き、必要なら proposal export を Markdown で整形する

要件:
- 出力品質ガード（§5-3）が Mixed-Scope 注記、フッター、見出し、根拠フォーマットを保証する
- chat と別に品質を安定化できる
- 後から再利用できる
- **保存先は §7 で確定済み**（`~/.daytrace/output/YYYY-MM-DD/`）

## 2. Canonical Timeline の固定

すべての生成物は共通の canonical timeline から派生させる。

時間帯スロット:

| スロット | 時間帯 |
|---------|--------|
| 深夜 | 00:00-06:00 |
| 午前 | 06:00-12:00 |
| 昼 | 12:00-14:00 |
| 午後 | 14:00-18:00 |
| 夜 | 18:00-24:00 |

目的:
- ダイジェスト、自分用日報、共有用日報、投稿下書きで時系列が食い違わないようにする
- 1 日の正史を 1 本に固定する

ルール:
- group の代表タイムスタンプ（最初のイベント）でスロットを決定する
- 自分用日報: スロット順で項目を並べる
- 共有用日報: インパクト順で並べるが、各項目にスロット名を付記する（例: 「[午後] テスト実装と品質強化」）
- 投稿下書き: スロット順で narrative を構成する
- ダイジェスト / 投稿下書き / 日報で、同一活動の時間帯記述が食い違わないこと

## 3. 事実・推測・語彙の統制

### 3-1. 事実と推測の分離
ルール:
- 自分用日報: 推測禁止
- 共有用日報: 推測禁止
- 提案: 推測禁止
- 投稿下書き: 推測可。ただし明示表現必須

推測可の表現例:
- 〜と見られる
- 〜が示唆された
- ログ上は〜
- 補助的に推定すると〜

禁止:
- 裏付けのない断定
- TDD 的にやった、提出直前だった、などの意図推定を断定調で書くこと

### 3-2. 禁止語と置換

**スコープ付き禁止語:** 出力タイプによって適用範囲が異なる。

全出力で禁止:
- 今日の重心 → 今日の中心タスク / 主な作業
- 実装密度の高い1日 → 具体的な成果数で表現
- 固定化 → 適用 / 保存
- ドラフト化 → 適用
- 寄り道 → 午前中は別作業を実施

日報・提案で禁止（投稿下書きでは推測マーカーとして許可）:
- 〜でしょう → 投稿下書きでのみ推測表現として使用可

### 3-3. 根拠データのサニタイズ
ルール:
- source 名は明示する
- 英語の生ログや会話タイトルをそのまま出さない
- 可能なら日本語の短い要約に変換する

例:
- `Configure my statusLine from my shell PS1 configuration`
  -> `シェルの statusLine 設定に関する AI 会話ログ`
- `ブラウザでの閲覧ログ`
  -> `chrome-history の閲覧ログ`

## 4. 生成物ごとの役割分離

### 4-1. 自分用日報
目的:
- 後で見返して思い出せること
- 正確な時系列
- 根拠が内部メモとして役立つこと

理想:
- 時系列順（canonical timeline のスロット順）
- 各項目は「何をした / なぜ / 結果」の 3 点
- 根拠は source 名を含む要約
- mixed-scope 注記は final artifact で確実に入れる

### 4-2. 共有用日報
目的:
- 第三者が短時間で成果と残課題を把握できること

理想:
- インパクト順（ただし各項目にスロット名を付記）
- 中心テーマ 1-2 本を前に出す
- 各項目は 1-2 文で完結
- 技術用語と内部事情を減らす
- 根拠は末尾の参考一覧にまとめる

### 4-3. 提案
目的:
- どの候補を今進めるべきか判断できること

理想:
- 候補ごとの差が分かる（出現回数、source 数、何が反復していたか、適用後の効果）
- 確度説明がコピペでない
- `何を / なぜ / どう進めるか` が分かる
- 内部状態語でなく、行動につながる文言にする
- chat には候補一覧の compact 表を出す（§6 の例外ルール）
- final artifact にはフル詳細を出す

### 4-4. 投稿下書き
目的:
- 読者が「今日何を作って何が分かったか」をすぐ理解できること

理想:
- 冒頭 1 文で主題が分かる
- 背景説明は 1-2 文に圧縮
- 本題と学びを前に出す
- 結びは感想文ではなく、学びか次のアクションで閉じる

### 4-5. セッション完了サマリ
目的:
- 何を生成し、何が次の一手かを短く伝えること

理想:
- 散文 3-5 行
- 箇条書きではなく、セッション全体の要約
- 日本語で統一
- 内部推論や英語混入なし

## 5. Mixed-Scope / Footer / 出力品質ガードの責務整理

### 5-1. Mixed-Scope 注記
方針:
- chat output では必須にしない
- final artifact の出力品質ガードが責任を持って挿入する

基本文:
- この日報は、1日のログと workspace ローカルの変更ログをもとに再構成しています。

### 5-2. 再構成元フッター
方針:
- final artifact で明示する
- chat output では省略または短縮
- source 名と scope を整形して統一表現にする

### 5-3. 出力品質ガード

**実装方針:** SKILL.md の生成ルール強化として実装する（Python スクリプトの後処理ではない）。LLM の自由テキスト出力に対して正規表現ベースの後処理は限界があるため、生成時点でルールを守らせる。

各 SKILL.md に追加するルールセット:

ガードでやること:
- §3-2 の禁止語チェック（スコープ付き）
- Mixed-Scope 注記の挿入
- フッター挿入
- §3-3 の根拠サニタイズ（英語生ログ除去）
- source 名の正規化
- 見出し構造の整形
- 内部状態語の排除（candidate_id, triage_status 等）
- [DayTrace:trace] タグの出力排除
- §3-1 の事実/推測ルール遵守

ガードでやらないこと（LLM 生成時ルールの範囲外）:
- テキスト切れ検出（§8 で別途対処）
- 出力の事後バリデーション

## 6. Chat Output Policy

chat は semi-final に留める。

原則:
- 全文をそのまま流さない
- 生成物の概要だけを伝える
- 意思決定に必要な部分だけを見せる
- final artifact がある場合は、そちらを正とする

例:
- 日報を生成しました（自分用 + 共有用）→ ~/.daytrace/output/2026-03-20/report-private.md
- 提案 5 件、今すぐ進められる候補は 2 件です
- 投稿下書き 1 本を作成しました → ~/.daytrace/output/2026-03-20/post-draft.md

chat に出すべきもの:
- source 収集結果の要約
- DayTrace ダイジェスト
- **proposal の候補一覧**（例外: 選択判断に必要なため compact 表で出す）
- selection prompt
- セッション完了の短い散文要約

proposal の compact 表の形式:
```
| # | 候補 | 種類 | 確度 | アクション |
|---|------|------|------|-----------|
| 1 | DayTrace 出力 UX ルール | CLAUDE.md | 高い | すぐに追加可 |
| 2 | 評価プロンプト | skill | 高い | /skill-creator で生成可 |
```

chat に出さなくてよいもの:
- 完全な Markdown 成果物本文
- 長い根拠一覧
- フッター全文
- 内部 trace

## 7. Artifact Storage Policy

final artifact の保存先と命名規約を固定する。

**保存先:** `~/.daytrace/output/YYYY-MM-DD/`

命名規約:
- `report-private.md` — 自分用日報
- `report-share.md` — 共有用日報
- `post-draft.md` — 投稿下書き
- `proposal.md` — 提案の詳細版（候補ごとのフル説明）

ルール:
- ディレクトリが存在しない場合は作成する
- 同日に複数回実行した場合は上書きする（最新が正）
- chat 出力時に保存先パスを明示する
- JSON source snapshot（`aggregate.json`）も同居可（任意）

---

## 8. Failure / Degrade Policy

出力失敗時の挙動も設計する。

テキスト切れへの対処:
- **根本対策:** final artifact をファイルに書き出すことで、chat のトークン上限やコンテキスト圧縮の影響を回避する
- chat には要約のみを出すため、chat 上でテキスト切れが発生しても成果物は無傷
- ファイル書き出し自体が失敗した場合は、chat に短い要約 + エラーメッセージを返す

その他の失敗:
- 一部 artifact だけ失敗したら、成功分だけ保存して失敗を明示する
- internal English leakage は §5-3 の出力品質ガード（生成時ルール）で防止する

## 9. Proposal UX の改善

### 9-1. 候補説明
候補ごとの差を明確にする。

改善方針:
- 出現回数
- source 数
- 何が反復していたか
- 何を適用すると次回どう楽になるか
を候補固有に出す

### 9-2. 用語
`ドラフト化` や `固定化` のような曖昧語は避ける。

### 9-3. Selection Prompt
現行:
- 今すぐ適用する候補を選んでください。番号を返すと適用フローに入ります。選ばなかった候補も次回以降の提案として保持されます。

（Wave 0 で修正済み）

### 9-4. 将来の複数選択
将来的には `1,3` のような複数選択を見据える。
ただし現時点では未対応でもよい。
仕様だけ先に意識しておく。

## 10. Optional Ask Policy

自律性を損なわずに品質を上げるため、ask の使い分けを定義する。

方針:
- 日報: ask なし完走
- 提案: ask なし完走
- 投稿下書き: optional に 1 回だけ方向性確認してよい
- 無回答でも fallback して完走

投稿下書きの ask は **chat output policy の例外ではない**: 方向性確認は chat に 1 文で出すだけであり、完全な成果物を chat に載せる理由にはならない。

投稿下書きの例:
- 技術寄り
- 振り返り寄り
- 発見寄り

ただし、
- 聞かなくても出せる
- 聞いた場合だけ質を上げる
が原則

## 12. 実装優先順

依存関係を考慮して再編した順序:

優先度高:
1. ~~バグ修正（固定化/ドラフト化の残存、Phase 5 箇条書き）~~ → Wave 0 完了
2. Canonical timeline ルールの追記（daily-report / post-draft SKILL.md）
3. 事実/推測/禁止語ルールの追記（各 SKILL.md）
4. ~~Selection prompt の修正~~ → Wave 0 完了
5. Artifact storage policy の確立（§7 の実装: ディレクトリ作成 + Write 出力）

優先度中:
6. Chat output policy の実装（Output Order/Skeleton 書き換え）— §7 の後
7. 出力品質ガードの SKILL.md 追記
8. 根拠 source 名の正規化
9. 投稿下書きの fact vs inference ルール明文化
10. Proposal 候補説明の差別化

優先度低:
11. 複数選択対応
12. 投稿下書きの optional style ask
13. Mixed-scope 注記の final artifact 移動（chat 圧縮完了後）
