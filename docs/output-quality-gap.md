# Output Quality Gap

このメモは、`daily-report` と `post-draft` の「理想的な状態」と「現状の出力」の差分を、`SKILL.md` 改善のための原稿として整理するためのもの。
ここでは実装詳細の正しさよりも、ハッカソン審査員や第三者が読んだ時の第一印象、理解速度、読み心地を重視する。

## 1. 一言まとめ

現状の出力は「証跡は強い」が「文章としてはまだ硬い」。
特に問題なのは、実装名・ファイル名・Issue 名・差分量・CLI 名が本文の前景に出すぎており、
本来前に出るべき

- 何に取り組んだのか
- なぜそれをやったのか
- 何が進んだのか
- 何がまだ残っているのか

よりも、「ログをうまく並べた感」が勝ってしまう点にある。

加えて、2026-03-19 の実出力からは、文章品質以前の **構造的な出力欠陥** も可視化された。
フェーズメタデータの混入（`[DayTrace] Phase 2: mode=両方 | ...`）、
本文の途中切れ、項目番号の飛びなどは、読者体験を根本から損なう。
これらは「文章が硬い」以前の問題であり、最優先で潰すべき。

また、低信頼度データ（Chrome 履歴・ワークスペースファイル活動など）の扱いが未整理であり、
「使う/使わない」の判断基準と使用回数制限がない。

理想は、証跡の強さを保ったまま、

- `report draft` は「短時間で状況が把握できる共有文」
- `post-draft` は「読者が流れを追える一次情報ベースの物語」

として読めること。

## 2. `report draft` の理想像

### 理想

理想的な `report draft`、特に共有用日報は、読者が 20-40 秒で以下を把握できる状態にある。

- 今日の主要テーマは何だったか
- 実際に進んだ成果は何か
- 残課題は何か
- 技術的な中身を知らなくても、前進が見えるか

文章の主語は「ファイル」や「関数」ではなく、「行動」「判断」「成果」であるべき。

たとえば理想は、

- `skill_miner_prepare.py を +65/-21 修正した`

ではなく、

- `パターン抽出の不安定さを減らすために、準備段階のロジックを修正した`

と読めること。

根拠は必要だが、本文の主役ではない。
本文は人間が読む文章、根拠はその後ろで支える証拠、という前後関係が必要。

### 理想的な読後感

- 技術者でなくても「今日はこれを進めた日だ」と分かる
- 技術者なら、必要な時だけ根拠から深掘りできる
- 実装の羅列ではなく、進捗報告として読める
- 書き手の頭の中の整理ではなく、読み手の理解順で並んでいる

## 3. `report draft` の現状

現状の `report draft` は、事実量と証跡密度は十分ある。
一方で、共有用として見ると次の問題がある。

### 3-1. 実装名が前に出すぎる

本文の中心に

- `observation contract`
- `skill_miner_prepare.py`
- `ISSUE-observation-contract-unification.md`
- `PR #3`
- `P1-P10`
- `+65/-21`

のような内部識別子が連続して出てくる。

これは開発当事者には意味があるが、第三者にとっては「何が進んだか」より先に「内部名の解読」が必要になる。

結果として、内容は分かるが、読む負荷が高い。

### 3-2. 根拠が強い代わりに、本文がログ寄りになっている

現状の根拠欄は、

- commit hash
- diff 行数
- 会話ログの原文
- Chrome 履歴の具体サイト名

まで近い粒度で出てくる。

これは証拠としては強いが、共有用本文に近接しすぎると、文章全体が「監査ログ」に寄る。

理想の共有用日報では、

- 本文は自然文
- 根拠は短く抽象化
- 必要なら末尾や注記で補足

の順であるべき。

### 3-3. 粒度が揃っていない

同じレベルで並んでいる項目の中に、

- 主要な技術判断
- バグ修正
- リポジトリ整理
- ハッカソン準備
- 私的な予定

が混在している。

自分用なら成立するが、共有用では「今日の主戦場」と「補助的な出来事」を分けた方が読みやすい。

現状は「全部正しく拾えている」が、「何が主で何が従か」の編集がまだ弱い。

### 3-4. 成果より作業量が見えやすい

`PLAN.md × 3 削除`
`+65/-21`
`design-notes/ 削除`
のような表現は、手を動かした量は伝わるが、読者にとっての意味は薄い。

共有用では

- 何を整理したのか
- その結果、何が分かりやすくなったのか
- 何のための整理だったのか

まで翻訳されて初めて価値として伝わる。

### 3-5. 共有文として少し“硬すぎる”

全体として、

- 事実は正確
- 証跡もある
- だが文章の温度が低い

という状態。

「今日どこが進んだか」が見えるより、
「今日どのログをどうまとめたか」が見えてしまう。

### 3-6. 構造的な出力欠陥（2026-03-19 実出力より）

今回の実出力では、文章品質以前に次の構造的な問題が確認された。

- **フェーズメタデータの混入**:
  `[DayTrace] Phase 2: mode=両方 | mode_source=default | item_count=29 | primary_evidence=mixed`
  のようなオーケストレーション内部情報が、ユーザー向け出力にそのまま表示されている。
  審査員がこれを見た場合、「デバッグ出力が残った未完成品」という印象を与える。

- **本文の途中切れ**:
  「ハッカソン提出に向�-0 追加〜」「DayTrace の�ーキテクチャ」「観測され�ding のまま」
  のように、文の途中で切断されている箇所がある。
  出力トークン制限、またはバッファの問題の可能性がある。

- **項目番号の飛び**:
  活動項目が 1, 2, 3, 5 と並び、4 が欠落している。

これらは「文章が硬い」以前の問題であり、最優先で対処すべき。

### 3-7. 低信頼度データの扱いが未整理

Chrome 履歴やワークスペースファイル活動など、信頼度の低いソースからのデータが、
高信頼度ソース（Git、AI 会話ログ）と同じ重みで本文に混入している。

現状の SKILL.md には `注記:` による inline 表記ルールがあるが、
「低信頼度データをいつ使い、いつ落とすか」の判断基準が明示されていない。

理想は、

- 低信頼度データは「裏付け」「補助」として使い、それ単独で主要根拠にしない
- 同じ低信頼度データを複数箇所で繰り返さない（一つの項目につき最大一回）
- 低信頼度データ単独で独立した活動項目を構成しない
- 高信頼度ソースと組み合わせたときにだけ採用し、「補助推定」であることを明示する

こと。使わないのではなく、「使い方と使用回数」を制御するのがポイント。

## 4. `report draft` の理想との差分

差分を短く言うと、

- 現状: `ログに忠実な進捗報告`
- 理想: `第三者が短時間で理解できる共有文`

である。

不足しているのは情報量ではなく、編集方針。
特に必要なのは次の変換。

- 実装名 → 行動レベルの表現
- 差分量 → 成果の意味
- 会話断片 → 判断の要約
- ログ列挙 → 主要テーマ中心の再構成

## 5. `post-draft` の理想像

### 理想

理想的な `post-draft` は、ログをなぞる文章ではなく、
「その日、何を直し、どこで詰まり、何に気づいたか」を一つの流れとして読める下書きである。

読者はまず

- 何がテーマの一日だったのか
- どこが面白いのか
- 何が学びだったのか

を掴み、その後で必要に応じて技術名に触れる。

つまり、理想の `post-draft` は

- 技術ログを narrative に変換している
- タイトルに「読んでみたい理由」がある
- 本文に転換点がある
- 学びが具体的で、かつ内輪すぎない

状態であるべき。

### 理想的な読後感

- 開発当事者でなくても流れが追える
- 技術的な話だが、読み物として成立する
- 「今日は何をしたか」ではなく、「今日は何が起きたか」になっている
- 最後に一つ、外に持ち出せる学びが残る

## 6. `post-draft` の現状

現状の `post-draft` は、中心テーマの選び方自体は悪くない。
`observation contract` 統合を主題に据えた判断も妥当。
ただし文章として見ると、まだ次の問題がある。

### 6-1. タイトルと導入が内輪寄り

現状タイトルの

- `DayTrace の心臓部`
- `observation contract`

は、すでに文脈を知っている読者には響くが、初見読者には少し閉じている。

「なぜそれが重要なのか」がタイトル単体では伝わりきらず、
読み出しの敷居がやや高い。

### 6-2. “背景” が説明文に寄りすぎる

背景段落は内容自体は正しいが、

- DayTrace の説明
- skill_miner_prepare.py の役割
- observation contract の乖離

を短時間に詰め込んでおり、読み味としては説明資料寄りになっている。

記事下書きとしては、

- 何が困っていたのか
- その困りごとが今日どう表面化したのか

から入った方が、流れが自然になる。

### 6-3. 技術識別子がそのまま残りすぎる

本文中に

- `ISSUE-observation-contract-unification.md`
- `skill_miner_prepare.py`
- `P9`
- `worktree_status snapshot`

のような語が出ると、読者はそこで一度止まる。

開発ログとしては正しいが、投稿下書きとしては
「意味が伝わる言葉」に一段変換したい。

### 6-4. 転換点はあるが、感情線がまだ弱い

現状でも

- 統合作業
- 別問題の浮上
- ハッカソン準備への視点移動

という流れはある。

ただ、それぞれがやや説明的につながっており、
「今日はここで視点が変わった」という narrative の起伏がまだ弱い。

もう少し

- 何を直していたつもりだったか
- どこで別の問題が見えたか
- なぜ夕方にハッカソン準備へ重心が移ったのか

が人の行動として読めると、文章が生きる。

### 6-5. 学びがやや一般化しすぎている

最後の

- `ドキュメントと実装の乖離は小さく見えても LLM の動作に直結する`

は方向として正しい。
ただし今のままだと、やや「良いことを言っている一般論」で止まりやすい。

理想は、

- 何がズレていたのか
- そのズレが何を不安定にしたのか
- だからどこを揃えるべきなのか

がもう半歩具体的に残ること。

### 6-6. 構造的な途中切れ（2026-03-19 実出力より）

`post-draft` でも本文の途中切れが確認されている。
タイトル・背景・気づきの各セクションが揃っていても、
途中の文が切断されていると読者の信頼は即座に失われる。

また、`post-draft` セクションの前後にフェーズメタデータ
（`[DayTrace] Phase 4: selected_group_id=group-002 | topic_tier=1 | ...`）
が挟まっており、narrative の没入感が途切れる。

## 7. `post-draft` の理想との差分

差分を短く言うと、

- 現状: `技術ログを丁寧に文章化した下書き`
- 理想: `読者が流れと学びを追える記事下書き`

である。

不足しているのは主題選定ではなく、読者向けの編集。
特に必要なのは次の変換。

- コンポーネント名 → 問題の意味
- Issue 名 → 取り組みのテーマ
- 調査メモ → 転換点
- 一般論の学び → 今日の出来事から出た具体学び

## 8. 両者に共通する本質的な問題

`report draft` と `post-draft` の両方に共通するのは、
「証跡を捨てられず、そのまま本文の近くに残しすぎる」こと。

その結果、

- 文章が実装寄りになる
- 日本語の行動表現より内部識別子が勝つ
- 読み手が理解する前に、書き手の作業メモが見えてしまう

という現象が起きている。

加えて、2026-03-19 の実出力からは、もう一つ根本的な問題が見えた。

- **構造的な出力品質**（途中切れ・メタデータ混入・番号飛び）が担保されていない
- **低信頼度データの使用ポリシー**が存在しない

証跡をそのまま出すか、編集して出すかの判断が曖昧なまま SKILL.md に書かれており、
LLM は「全部正しく出す」方に倒れやすい。
結果として、本来は裏方に退くべき実装名・commit hash・フェーズ表示が前景に出る。

つまり問題はデータ不足ではなく、

- 何を本文に出すか
- 何を根拠へ退避するか
- どの語彙まで抽象化するか

の編集ポリシー不足にある。

## 9. `SKILL.md` で直すべき方向

この差分から逆算すると、`SKILL.md` では少なくとも次を強化する必要がある。

### 9-1. 本文語彙の変換ルール

- `aggregate.py を修正` ではなく `ログ集約処理を改善`
- `skill_miner_prepare.py` ではなく `パターン抽出の準備段階`
- `ISSUE-xxx.md` ではなく `観測コントラクト統合作業`

のように、本文ではまず意味を出し、固有名は必要時だけ補助に回す。

### 9-2. 根拠の短文化

根拠は

- commit hash
- 差分行数
- 生の会話引用

をできるだけ直接出さず、

- `Git の修正履歴`
- `AI 会話ログでの設計検討`
- `関連ドキュメントの更新履歴`

レベルまで要約してから出す。

### 9-3. 主従の編集

その日の中心テーマを 1-2 本に絞り、
他の出来事は

- 補助的な進行
- 周辺整理
- 次フェーズ準備

として扱う。

全部を同じ重さで並べない。

### 9-4. narrative の転換点ルール

`post-draft` では特に、

- 問題提起
- 修正着手
- 途中で見えた別問題
- 視点の転換
- 最後の学び

のどこが今日の転換点だったかを、本文で明示的に拾う必要がある。

### 9-5. “読者が読む順” の優先

ログ上の時系列や内部識別子の正しさよりも、

- まず全体像
- 次に中心テーマ
- その後に根拠

の順で理解できる構成を優先する。

### 9-6. 低信頼度データの使用ポリシー

現状の SKILL.md には `注記:` の表記ルールはあるが、
「低信頼度データを使う / 使わない」の判断基準が欠けている。
以下のポリシーを明示する。

使う場面:

- 高信頼度ソース（Git、AI 会話ログ）の裏付けとして、活動の文脈を補強できるとき
- 時間帯や行動の切り替わり（例: 午後に調査系へ移行した）を示す間接証拠として

使わない場面:

- 低信頼度データ単独で活動項目を構成するとき
- 同じ低信頼度データが複数箇所で繰り返し引用されるとき（使うなら一回まで）
- Chrome 履歴の個別ページ名やローカルパスがそのまま本文に出るとき

原則:

- 低信頼度データは「一回・補助・裏方」
- 高信頼度ソースとペアにして初めて採用する
- 本文には意味だけ出し、生データは根拠に退避する

## 10. まとめ

現状の DayTrace 出力は、

- 事実の強さ
- 根拠の豊富さ
- 主要テーマの検出

はすでにかなり良い。

課題は、そこから先の「読み物としての編集」にある。

`report draft` は、共有用の進捗文としてもう一段やわらかく、早く理解できる必要がある。
`post-draft` は、技術ログの文章化から一歩進んで、読者が流れと学びを追える narrative に寄せる必要がある。

したがって次にやるべきことは、projection や source を増やすことではなく、
`SKILL.md` 上で

- 本文語彙
- 根拠の出し方
- 主従の編集
- narrative の転換点

を明示的にルール化することである。

## 11. 実装プラン方針

この差分分析を、SKILL.md の具体的な改善に移すための実装方針を以下に示す。

### 11-1. 対処の優先順位

優先順位は「読者体験への影響度」で決める。

1. **構造的欠陥の解消**（最優先）
   - フェーズメタデータをユーザー向け出力から除外する
   - 途中切れの原因を特定し、出力長制御を入れる
   - 項目番号の連続性を保証する

2. **低信頼度データの使用ポリシー追加**
   - SKILL.md に「使う/使わない」の判断基準を明記する
   - 「一回・補助・裏方」の原則を output formatting ルールに組み込む

3. **本文語彙の変換ルール強化**
   - 実装名 → 行動レベル表現の変換テーブルを examples として SKILL.md に追加
   - 根拠の短文化ルールを明記（commit hash → `Git の修正履歴`）

4. **主従の編集ルール追加**
   - 中心テーマ 1-2 本 + 補助的な出来事、という構成ルールを明記
   - 自分用/共有用で粒度差を強めるガイダンスを追加

5. **narrative 転換点ルール**（post-draft 固有）
   - 「どこで視点が変わったか」の明示を SKILL.md に追加
   - 学びを「今日の出来事から出た具体学び」に寄せるルールを追加

### 11-2. 変更対象

- `plugins/daytrace/skills/daily-report/SKILL.md` — 語彙変換、根拠短文化、主従編集、低信頼度データポリシー
- `plugins/daytrace/skills/post-draft/SKILL.md` — narrative 転換点、タイトル生成、低信頼度データポリシー
- `plugins/daytrace/skills/daytrace-session/SKILL.md` — フェーズメタデータの出力制御（出力パイプライン側）
- `plugins/daytrace/skills/daily-report/references/` — 語彙変換テーブル、output examples の追加

### 11-3. 進め方

1. まず 11-1 の優先度 1（構造的欠陥）を潰す — これは品質以前の問題
2. 次に 11-1 の優先度 2-3（低信頼度データ + 語彙変換）を SKILL.md に反映する
3. その後、実出力を再生成して改善を確認する
4. 優先度 4-5 は確認結果を見て判断する
