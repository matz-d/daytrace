更新注記（2026-03-18）

この文書の本文には、`skill-miner` の carry-forward loop を閉じる前の監査メモが残っている。現コードを正とする場合、少なくとも以下は本文よりこの注記を優先する。

- テスト失敗前提は obsolete。現行の scripts test suite は `python3 -m pytest plugins/daytrace/scripts/tests/ -v` で **271 passed**
- Decision log は write-only ではない。`skill_miner_prepare.py` は `load_latest_decision_states()` で JSONL を読み、`carry_forward=false` を suppress し、`reject` は resurface 条件で再評価する
- `daytrace-session` は Phase 3 で `--decision-log-path` / `--skill-creator-handoff-dir` を明示し、proposal を session-specific temp file に保存してから `skill_miner_decision.py` と `skill_miner_proposal.py --user-decision-file` で writeback する
- `skill_miner_decision.py` が追加され、`adopt + completed` のみ suppress 側に倒し、未完了 adopt は `defer` 相当に正規化する
- store migration は将来課題だが、現フェーズの user decision writeback loop 自体は closed になっている
- `test_e2e_pipeline.py` は `--reference-date 2026-03-15` を固定し、fixture-backed E2E の時計依存を除去済み
- `skill-miner` SKILL.md の decision log contract から future-work / 未実装に読める文言を外し、現フェーズの挙動に揃えた
- `near_match_dense` は通常 near-match を research input、`complete_link_guard` 由来のみを ready-blocking ambiguity とみなす設計だとコードコメントと本文に明文化した。提出直前のロジック変更は見送っている

  2. 重大リスク（Critical）

  ✅ C-1: テスト失敗リスクは現行コードでは解消済み

  現時点の確認結果:

   python3 -m pytest plugins/daytrace/scripts/tests/ -v
   → 271 passed

  原因の構造：

  skill_miner_proposal.py の実行フローが golden fixture 
  の前提と乖離している。

   1. build_proposal_sections() 内で markdown を1回生成（context_file なし）
   2. persist_skill_creator_handoffs() でファイルを書き、context_file 
  をパスとして candidate に注入
   3. proposal.py の main() が markdown を再度生成し上書き（今度は 
  context_file パスが含まれる）

  結果として出力 markdown に:

      handoff: 
  /Users/makotomatuda/.daytrace/skill-creator-handoffs/20260318T...long.json
      公式 handoff: /skill-creator prepare-report-...

  が出るが、golden fixture には handoff: 行がない。

  追加問題: この golden fixture テストは本質的に 
  non-reproducible。context_file
  には実行日時が含まれるため、golden に path を含める限り永遠に一致しない。

  見るべきコード:

   - skill_miner_common.py:2573-2576 (context_file の markdown 出力)
   - skill_miner_proposal.py:329-339 (handoff persist → markdown 
  再構築の二重build)
   - tests/test_skill_miner_proposal.py:33-43 (_render_markdown_from_fixture
   の実装)

  現状の読み替え:
  この項目は historical note として扱う。現コードを説明する文脈では、
  「tests are green」を前提にしてよい。

  ✅ C-3: near_match_dense は既知の設計判断として文書化済み

  near_match_dense フラグは「near_match_count >= 2 かつ 
  "complete_link_guard" in
  near_match_reasons」の AND 条件で立つ。

  しかし prepare.py で、通常の近接スコア（0.45–0.55）で登録された near_match は
  reason なし（≠ "complete_link_guard"）で記録される。

  結果として、complete_link_guard 由来以外の near_match が何件積み上がっても
  near_match_dense フラグは立たず、他の blocking flag がなければ ready に残れる。

   # prepare.py:1237-1239
   register_near_match(left_index, right_index, score, 
  reason="complete_link_guard")  # complete-link由来
   elif CLUSTER_NEAR_MATCH_THRESHOLD <= score < CLUSTER_MERGE_THRESHOLD:
       register_near_match(left_index, right_index, score)  # reason=None → 
  near_match_dense が立たない

  現コードではこれは意図的な仕様として扱う。

  設計意図:

   - 通常 near-match は「似た別候補があるので追加調査に回す」ための research input
   - `complete_link_guard` 付き near-match は「pairwise では merge できそうだが、
  component 全体では bridge merge になるので止めた」ことを示す強い曖昧性シグナル
   - `near_match_dense` は後者が split signal と併発した時だけ ready-blocking にする

  対応:

   - `skill_miner_prepare.py` と `skill_miner_common.py` にコメントを追加し、この意図をコード上で説明した
   - 提出直前は clustering / triage の振れを避けるため、ロジック変更は行っていない

  判断:

  ハッカソン提出前の issue としては close してよい。将来、false-ready が実データで
  問題化した場合に限って、通常 near-match も blocking に含めるかを再検討する。

  3. 重要だが提出後でよい改善

  I-1: markdown 二重生成
  build_proposal_sections() 内と proposal.py main() の2箇所で markdown
  を生成し後者が前者を上書きする。中間 markdown
  は捨てられる。これは設計の不整合で、将来の修正時にバグを生みやすい。今は副
  作用がないが技術的負債。

  I-2: hook/agent の next_step_stub が skeletal
  skill 分類では skill_scaffold_context + skill_creator_handoff +
  ファイル永続化まで実装されている。hook/agent は next_step_stub 
  の1オブジェクトのみで、trigger_event/
  action_summary/role_summary が手薄。デモで hook 
  候補が出た場合、次のアクションが「次セッションへ送る」だけになる。

  I-3: daytrace-session の proposal コマンドに永続化フラグが未指定
  SKILL.md の Phase 3 に書かれているコマンドは --prepare-file と 
  --judge-file のみ。実際には 
  --decision-log-path と --skill-creator-handoff-dir 
  が未指定のため、実行のたびにデフォルト 
  ~/.daytrace/ 以下に書き込む。副作用があることが SKILL.md から読めない。

  I-4: user_decision="adopt" + skill の store 
  フラグが未実装と自己申告されている
  SKILL.md line 304: 将来 store に adopted フラグを立てる（未実装）
  という記述が残っている。「未実装」という文字列を提出物の仕様書に残すのは 
  avoid すべき。「現フェーズでは対象外」「v2
  で実装予定」といった表現に置き換えるか、節ごと削除する。

  I-5: 用語の不統一 

   - SKILL.md では「Skill Scaffold Draft」と呼んでいる概念が、コードでは 
  skill_scaffold_context と skill_creator_handoff に分かれている。
   - references/ で使われている用語と SKILL.md 
  の見出し語に細かいズレがある。

  4. 二重実装・責務衝突・概念のズレ

  D-1: markdown 生成の責任が 2 箇所に分散
  skill_miner_common.py:build_proposal_sections() が markdown を返す（内部 
  helper
  用）、skill_miner_proposal.py:main() が最終 markdown を上書きする（CLI
  出力用）。この分離は明示されていないため、直接 build_proposal_sections() 
  を呼ぶコード（テストや
  daytrace-session が内部で使う場合）は context_file の含まれない古い 
  markdown を取得する。

  D-2: build_candidate_decision_stub で生成される stub と 
  apply_user_decisions の ID
  照合
  decision_log_stub は candidate_id 
  でキー管理されているが、apply_user_decisions() では 
  candidate.get("candidate_id") or candidate.get("packet_id") と2つの field 
  を
  OR で読む（proposal.py:173）。decision_log_stub 側は candidate_id のみ（
  skill_miner_common.py:2041-2046）。packet_id しか持たない unclustered 
  packet の
  stub では user_decision overlay が当たらない可能性がある。

  D-3: daytrace-session vs 個別 skill の責務境界
  SKILL.md に「個別 skill 
  の出力フォーマットや品質基準を上書きしない」とある一方、Phase 2
  の「共有用追加生成の条件（total_groups >= 5）」は daily-report SKILL.md
  の定義外の条件。orchestration 側が個別 skill 
  のトリガー条件を勝手に定義している。

  5. デモ視点の最終助言

  見せれば強いもの：

   - 同じ作業パターン（例：pytest 実行 → git commit → findings 
  を記録、を複数セッションで繰り返したもの）が cluster 
  として浮かび上がり、CLAUDE.md に即適用できる diff preview 
  まで自動生成されるシーン
   - source が1-2本しかなくても graceful degrade 
  で形になった日報が出るシーン（「壊れない」信頼感）
   - 0候補時の enriched output（観測サマリ・成長兆候）— 
  これが「空振りでも価値がある」設計思想を体現する

  見せない方がいいもの：

   - hook/agent 候補が出るシーン（next_step 
  が薄く「次セッションへ」で終わる）
   - clustering が oversized になって ready 0 件になるシーン
   - store migration まで質問を広げられるシーン（現実装は JSONL loop が正）
