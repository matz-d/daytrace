  2. 重大リスク（Critical）

  🔴 C-1: テスト1件が必ず失敗する（最優先）

   FAILED test_skill_miner_proposal.py::ProposalCLITests::test_cli_markdown_
  matches_golden_fixture

  原因の構造：

  skill_miner_proposal.py の実行フローが golden fixture 
  の前提と乖離している。

   1. build_proposal_sections() 内で markdown を1回生成（context_file なし）
   2. persist_skill_creator_handoffs() でファイルを書き、context_file 
  をパスとして candidate に注入
   3. proposal.py の main() が markdown を再度生成し上書き（今度は 
  context_file パスが含まれる）

  結果として出力 markdown に:

      handoff: 
  /Users/makotomatuda/.daytrace/skill-creator-handoffs/20260318T...long.json
      公式 handoff: /skill-creator prepare-report-...

  が出るが、golden fixture には handoff: 行がない。

  追加問題: この golden fixture テストは本質的に 
  non-reproducible。context_file
  には実行日時が含まれるため、golden に path を含める限り永遠に一致しない。

  見るべきコード:

   - skill_miner_common.py:2573-2576 (context_file の markdown 出力)
   - skill_miner_proposal.py:329-339 (handoff persist → markdown 
  再構築の二重build)
   - tests/test_skill_miner_proposal.py:33-43 (_render_markdown_from_fixture
   の実装)

  最小修正案: テストで --skill-creator-handoff-dir に tempdir 
  を指定し、markdown 比較時に 
  handoff: /... 行を除外するか、golden fixture を --update-golden 
  で再生成した後、path
  行を除いた形に正規化する。

  3. 重要だが提出後でよい改善

  I-1: markdown 二重生成
  build_proposal_sections() 内と proposal.py main() の2箇所で markdown
  を生成し後者が前者を上書きする。中間 markdown
  は捨てられる。これは設計の不整合で、将来の修正時にバグを生みやすい。今は副
  作用がないが技術的負債。

  I-2: hook/agent の next_step_stub が skeletal
  skill 分類では skill_scaffold_context + skill_creator_handoff +

  の1オブジェクトのみで、trigger_event/
  action_summary/role_summary が手薄。デモで hook 
  候補が出た場合、次のアクションが「次セッションへ送る」だけになる。

  I-3: daytrace-session の proposal コマンドに永続化フラグが未指定

  --judge-file のみ。実際には

  4. 二重実装・責務衝突・概念のズレ

  D-1: markdown 生成の責任が 2 箇所に分散
  skill_miner_common.py:build_proposal_sections() が markdown を返す（内部 
  helper
  用）、skill_miner_proposal.py:main() が最終 markdown を上書きする（CLI
  出力用）。この分離は明示されていないため、直接 build_proposal_sections() 
  を呼ぶコード（テストや
  daytrace-session が内部で使う場合）は context_file の含まれない古い 
  markdown を取得する。

  D-2: build_candidate_decision_stub で生成される stub と 
  apply_user_decisions の ID
  照合
  decision_log_stub は candidate_id 
  でキー管理されているが、apply_user_decisions() では 
  candidate.get("candidate_id") or candidate.get("packet_id") と2つの field 
  を
  OR で読む（proposal.py:173）。decision_log_stub 側は candidate_id のみ（
  skill_miner_common.py:2041-2046）。packet_id しか持たない unclustered 
  packet の
  stub では user_decision overlay が当たらない可能性がある。

  D-3: daytrace-session vs 個別 skill の責務境界
  SKILL.md に「個別 skill 
  の出力フォーマットや品質基準を上書きしない」とある一方、Phase 2
  の「共有用追加生成の条件（total_groups >= 5）」は daily-report SKILL.md
  の定義外の条件。orchestration 側が個別 skill 
  のトリガー条件を勝手に定義している。

  として浮かび上がり、CLAUDE.md に即適用できる diff preview 
  まで自動生成されるシーン
