# TODO Hackathon Pre

DayTrace をハッカソン提出レベルまで仕上げるための最終 TODO。
目的は単なるバグ修正ではなく、以下の 3 点を審査員に伝わる形で成立させること。

- Reliability: 壊れず最後まで走る
- Autonomy: 一度任せたら追加指示なしで進む
- Growth: 使うほど Claude Code の作法を学習し、次の改善につながる

審査基準への対応:

- 自律性 40%: 統合フロー、自己判断、fail-soft
- クオリティ 35%: 安定動作、分かりやすい出力、再現可能デモ
- インパクト 25%: 日報生成に留まらず、Claude Code 自体を育てる価値

## 0. ゴール定義

ハッカソン時点で見せる完成像:

- `今日の振り返りをお願い` の一言で主要フローが完走する
- 利用可能ソースを自分で判断して、欠損時も継続する
- 日報、反復パターン提案、必要なら追加調査、投稿下書きまで自律的に進む
- 途中判断をユーザーに説明できる
- ローカルで安定して再現できる

非ゴール:

- 大規模リファクタリング
- 完全なアーキテクチャ再設計
- 将来向け抽象化のやり切り

## 2. 審査で勝つための自律フロー追加

### 2-1. 統合 Skill を作る

新しい統合 skill の仮名:

- `daytrace-session`
- もしくは `daytrace-review`

要件:

- [x] 1 回の依頼で `収集 -> 日報 -> パターン提案 -> 追加調査 -> 投稿下書き` まで進める
- [x] 既存 3 skill を呼び分ける orchestration role を担う
- [x] ユーザーが 3 回別々に呼ばなくても完走する

### 2-2. 自己判断ログを出す

統合フローで必ず見せる判断:

- [x] 利用可能ソース数とスキップ理由
- [x] どのソースで続行するか
- [x] 共有用日報も出すかどうか
- [x] 追加調査を走らせるかどうか
- [x] 投稿下書きまで生成するかどうか
- [x] `CLAUDE.md` 適用候補があるかどうか

出力例:

- `4ソースが利用可能でした。chrome-history は権限不足のためスキップし、残りで続行します。`
- `活動グループが多いため、共有用の要約も自動生成します。`
- `1件は追加調査が必要だったため、関連 session を自動で確認しました。`

### 2-3. 自動判断ルールを最低限固定する

- [x] `groups` が一定数以上なら共有用日報も生成
- [x] `proposal_ready` 候補があるなら提案を表示
- [x] `needs_research` 候補があるなら detail + judge を自動実行
- [x] AI + Git 共起が強ければ `post-draft` まで生成
- [x] ソースが一部欠けても成功 source だけで続行

完了条件:

- 1 回の依頼で「自律性」を目で見て理解できる
- 審査員が付きっきりで指示しているように見えない

### 2-4. #3 の前に固定すること

今回の通し確認から、デモ品質の最適化に入る前に以下を固定しておく。

- [ ] `skill-miner` の空振り時 UX を固定する
  - `提案成立なし` でも価値が伝わる返しにする
  - 例: `split-aware` な再提示、または `次に何を改善すると候補化しやすいか` を 1 文で返す
- [ ] 統合 skill の最終出力順を固定する
  - `日報 -> skill-miner -> 必要なら追加調査 -> post-draft` を基準順にする
  - `提案成立なし` の場合でも全体の narrative が途切れないようにする
- [ ] デモ用の golden run を 1 本持つ
  - 統合 skill が同じ骨格で 3 回連続完走することを確認する
  - 必要なら fixture 相当の store / workspace で e2e 手順を固定する

理由:

- 今の実装は個別 skill としては成立しているが、統合デモとして見た時に `skill-miner` が `提案成立 0 件` で終わると自己改善ループが弱く見える
- #3 は演出最適化なので、その前に「統合フローが毎回どう着地するか」を固定しておく方が安全

完了条件:

- 統合 skill の着地が毎回同じ順で説明できる
- `skill-miner` が有力候補なしでも価値を損なわない
- #3 では演出とデモデータ固定に集中できる

### 2-5. ハッカソン前の `skill-miner` 強化

- [ ] ハッカソン前に `skill-miner` を強化する

以下は「やるべきタスク」の分解ではなく、最低限押さえるべき強化アイディアとして扱う。

- `提案成立 0 件` でも、なぜ 0 件だったかと次の打ち手が分かる出力にする
- oversized cluster は通常提案に流さず、split-first で見せる
- judge が返した `split_candidate` を、単なる保留ではなく暫定候補として見せられるようにする
- `CLAUDE.md / skill / hook / agent` の分類が `TBD` で終わらないようにする
- 統合フローの中で `skill-miner` が空振りせず、自己改善ループの役割を説明できる状態にする

狙い:

- `skill-miner` を「候補が出るかもしれない機能」ではなく、「弱い時も理由と次の一手を返す機能」にする
- 提案品質の弱さを隠すのではなく、正直に可視化しつつ価値に変える

## 6. 発表での見せ方

今回の修正は単なるバグ修正ではなく、以下の進化として話す。

### Phase 1

- ローカル証跡を集める
- 日報と投稿下書きを自動生成する
- 繰り返し作業を見つける
- `CLAUDE.md` などに定着候補を提案する

### Phase 2

- 反復作業を継続学習する
- 自分の作法を agent / skill / hook に変換する
- Claude Code を個人に最適化された実働環境へ育てる

使える説明文:

`今回の提出は Phase 1 です。まずはローカル証跡を自律収集し、日報・提案・下書きまで完走します。Phase 2 では、そこからさらに一歩進めて、繰り返し作業を学習し、Claude Code 自体をユーザーごとに育てていくエージェントに発展させます。`
