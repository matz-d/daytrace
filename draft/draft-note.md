# 自分を記録するツールを作ったら、自分が記録されていた――Claude Code plugin「DayTrace」制作記

※本記事は、AIエージェント ハッカソン 2026 関連の取り組みとして、[AGIラボ](https://chatgpt-lab.com/) のマガジン寄稿用に執筆しました。

---

## 1. はじめに――「5日分」しかないのに、なぜか3週間が見えた

記事を書くために `~/.daytrace/output/` を開いたら、**たった5日分**しか入っていませんでした。開発途中からローカル保存に切り替えたから、仕方ないんですが……正直「材料、足りなくね？」って一瞬思いました。

でも Claude Code に「このフォルダの内容を読んで、記事のリファレンス作って」と一言頼んだら、**skill-miner の品質問題にハマった日、marketplace に公開した日、最終日のクラスタリング改善**みたいな「要所」が、ちゃんと並んで返ってきたんです。

**自分が作っているツールが、自分の作業を記録していた**――説明ばかりしていた設計が、ここで初めて体験として刺さりました。

この記事では、DayTrace の概要、開発で一番しんどかったところ、そして「エージェントっぽさ」をどう解釈したかを、できるだけ現場の言葉でまとめます。

---

## 2. DayTrace って何をするの？

DayTrace は **Claude Code のプラグイン**です。`/daytrace-session` と打つと、だいたい次をまとめてやります。

* **収集**：Git / Claude / Codex / Chrome / ファイル変更などから、その日の証跡を取る  
* **日報**：自分用・共有用の下書きをローカルに保存  
* **投稿下書き**：1日の流れを、読者向けの文章に寄せる  
* **パターン提案（skill-miner）**：履歴の反復を見て、CLAUDE.md / skill / hook / agent への候補を出す  

インストールは `claude plugin marketplace add matz-d/daytrace-plugin` → `claude plugin install daytrace` で済ませる想定。**設定不要・外部送信なし**に寄せています。

日報の各項目には「根拠」（どのログから来たか）が付きます。思い出せるだけじゃなくて、**どのツールの痕跡から復元したか**が見えるのが、手書き日報との違いです。

同じローカル証跡を、**日付軸（今日なにした？）**と **スコープ軸（最近ずっと同じことしてない？）**の2ルートで使い分ける設計にしています。振り返りと改善提案を、一本のデータから出したい、という意図です。

## 3. なぜ作ったか――「ログ、もうそこにない？」

きっかけは、Google Workspace と Claude Code / Codex をつないでいる**自分の業務ハブリポジトリ**でした。Codex に「日報つくって」と頼んだら、Chrome の履歴やローカルの変更履歴まで、思った以上に細かく取れた。

**「毎日の証跡って、もう端末側に溜まってるじゃん」**  
あとは束ねて、意味のある形に変換するだけ――DayTrace の中心はそこにあります。

ローカル完結は、カッコつけというより実務判断です。トークン運用と「入れたら動く」を両立しにくいので、**Calendar や Slack は読めない代わりに、インストールの摩擦を減らす**方向を選びました。出力先や体裁も、人が自分の運用に寄せられる余地を残しています。

---

## 4. 開発でいちばんキツかったところ（skill-miner の品質）

いちばん苦しかったのは **skill-miner** です。履歴から「あなた、こういう繰り返ししてますよ」と提案する機能なんですが、最初は**ほとんどのセッションが同じパターンに吸い込まれる**状態になってしまいました。

たとえば「コードを調べて直す」みたいな、誰でもやる形が支配的になると、**本当に自分固有の繰り返し**が埋もれる。すると「件数は出てるのに中身が薄い」提案が並ぶ。マジで頭を抱えました。

ここで方針を切り替えました。**提案数は固定しない。0件でも正常。強い候補がないなら、正直にそう言う。**  
「多さ」より「妥当性」を優先する――品質問題を通して、設計に組み込んだコアです。

---

## 5. 「一度命じたら任せろ」と、うちの「止まらない」解釈

ハッカソンのテーマは「一度命じたら、あとは任せろ」でした。DayTrace の自律性は、ちょっと言い方を変えています。

**「最後まで進めること」**です。ソースは毎日そろいません。Chrome は権限の壁があるし、Codex を触らない日もある。それでも、取れた分だけで日報まで出す。**壊れないこと**を優先するタイプの自律、と捉えています（いわゆる graceful degrade 的な話です）。

私は普段から **HITL（人が最終判断する）** を重視しているので、自動投稿で終わらせず、**材料を揃えて人が承認する**スタイルに寄せています。`skill-applier` も diff 確認つき、というのはその延長線です。

---

## 6. 自分用に広げるなら――fork しないカスタム

もし「自分の業務フローに寄せたい」となったとき、私は **DayTrace 本体を fork しない**方針にしています。やっているのは、業務ハブのリポジトリに **薄い wrapper を一枚かぶせる**だけ。Calendar や Drive、Obsidian の入口を足したり、1日の終わりに計画と実績を突き合わせるフローを足したり、**自分固有の文脈だけをローカル側に寄せる**イメージです。

共通の芯は本家プラグインに任せて、アップデートの恩恵を取り込みやすいまま、運用だけ深くする――この分け方は、普段から「人と AI の役割を設計する」仕事をしている自分の肌感にも合っています。

---

## 7. ハッカソンの話（正直モード）

一次審査を通ったときは「本当に自分？」って感じでした。デモは無音ハイライトを繋いだだけで、5分発表で「使うほど環境が育つ」芯を伝えきるのは難しかった。**入賞はできなかった**のですが、その時点でもう頭の中は「じゃあ記事で補完しよう」でした。

結果より残ったのは、**「作った本人が、ツールのユーザーでもある」**という地味な実感です。

---

## 8. まとめ

DayTrace は完成形ではありません。分類精度も、まだ人の判断が要る場面はあります。それでも、

* 証跡はすでにローカルにある  
* 毎日使うなら、欠けたソースがあっても最後まで形にする  
* 提案は「盛る」より「正直」  

この3つは、ハッカソン後もブレない軸にしたいです。

詳細は README やプラグイン側を見てもらうのが早いです。**フィードバックは X / GitHub まで**（`github.com/matz-d/daytrace-plugin`）。

あなたの証跡も、たぶんもうそこにあります。
