# DayTrace 設計思想（記事参照用）

## 中心的なアイデア：「証跡は、すでにそこにある」

DayTrace の出発点は「毎日使っている道具が記録を残している」という観察。
Git のコミット、Claude のセッションログ、Codex の履歴、Chrome の閲覧履歴——
これらはすでにローカルに存在する。それを「束ねて、意味のある形に変換する」だけでいい。

---

## 設計判断 ①：なぜ「ローカル完結」か

OAuth やクラウド API に依存するソースはコアに含めない判断をした。

**理由**: トークン管理と「設定不要」は同時に満たせない。

SaaS への自動投稿・テンプレ差し込みは別スキルや手動に任せ、
DayTrace は **ローカルに読める成果物まで** を責務とする。

結果として：インストール後、追加設定ゼロで動く。

---

## 設計判断 ②：なぜ「2つの軸」か

同じローカル証跡を、用途に応じて 2 つのルートで処理する。

### date-first（日付軸）
- 「今日何をしたか」を日報・投稿下書きに再構成
- 報告日ルール：ローカル時刻 06:00 未満は前日の暦日として扱う（夜ふかしセッション対応）
- 出力: `report-private.md` / `report-share.md` / `post-draft.md`

### scope-first（スコープ軸）
- 「最近ずっと同じことをしている」パターンを 7〜30 日窓で抽出
- workspace-local（このリポ向け）と global-personal（個人横断）を別軸で区別
- 出力: 提案リスト → CLAUDE.md / skill / hook / agent に適用可能

---

## 設計判断 ③：なぜ「提案に根拠と確信度」か

**LLM の出力をそのまま適用させない**、という方針。

各提案には：
- `evidence`：どのセッションで何回出現したか
- `confidence`：候補単位の確度（表示は「確度: 高い / 中程度 / まだ弱い」）
- `proposal_ready`：そのままドラフト化できるかどうか

ユーザーが承認するかどうかを判断できる状態にして初めて適用する。
見送った提案もログに残り、証跡が蓄積されれば自動で再浮上する。

---

## 設計判断 ④：なぜ「Graceful Degrade」か

5 つのソースが全部揃うことはめったにない（Chrome は権限が要る、Codex を使っていない日もある）。

**ソースが欠けても止まらず、取得できたデータだけで最後まで進む。**

「壊れない」信頼感が、毎日使えるツールの条件だと考えた。

---

## ハッカソン審査基準へのアプローチ

### 自律性 —「質問しない」ではなく「最後まで進む」
- 一度頼むと、収集・日報・下書き・提案まで自律完走
- 人の判断を仰ぐのは、共有範囲の確認・適用の承認など **影響の大きい決定だけ**

### クオリティ —「根拠のある提案」
- date-first / scope-first の 2 軸で同じデータを用途別に再構成
- 提案には evidence + 確度を付け、LLM 出力の信頼性を担保
- チャットはコンパクトな一覧、詳細は artifact（`proposal.md`）で使い分け

### インパクト —「使うほど、環境が育つ」
- 提案は CLAUDE.md / skill / hook / agent への適用候補として返る
- 見送った提案も decision log に残り、証跡が蓄積されれば再浮上
- 使い続けるほど、反復作業が自動化され、開発環境が自分仕様に育っていく
