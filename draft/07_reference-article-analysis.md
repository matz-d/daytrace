## 見出し構成（h2/h3 レベル）

```
前書き
ハッカソンについて
プロダクト紹介
  背景
  解決策
  動作（Case 1〜3）
  本プロダクトの優位点
  今後の展開
戦略
開発
  技術スタック
  手順（①〜④ の番号付きTips）
発表
最後に
```

**特徴**：「プロダクト紹介」が記事の前半を占め、「どうやって作ったか（戦略・開発）」は後半。
先に「何を作ったか・なぜすごいか」を伝えてから「どう実現したか」に進む順序。

## 冒頭フック（前書きの構造）

```
「非エンジニア医師が、ソロで準優勝」
——通常ハッカソンは複数のエンジニアで挑むものですが、
初参加の研修医である私が一人で挑戦し、準優勝を獲得しました。
```

- **意外性を一文目で宣言**（非エンジニア × ソロ × 準優勝）
- すぐ「前半は〇〇、後半は〇〇」と構成を予告→離脱防止
- 読者想定を明示：「ハッカソンに興味のある方 / 開発で悩む非エンジニアの方 / エンジニアの方」

## トーン・文体

| 要素 | 内容 |
|------|------|
| 一人称 | 「私」「自分」 |
| 文末 | です・ます調（丁寧、親しみやすい） |
| 読者への語りかけ | 「〜の方も多いのではないでしょうか」「〜が重要です」 |
| 断定の度合い | ほどよく断定（「重要」「必要」）。但し根拠を必ず付ける |
| 専門用語 | 使うが、即座に補足説明（例：「トリアージとは：〜」） |
| 感情・主観 | 「とても印象に残っていた」「雲の上の存在」など個人的エピソードを挿入 |

## 情報粒度の特徴

### 技術スタックは具体的に列挙
```
Frontend: Streamlit
Backend: FastAPI
ML/LLM: Sentence-Transformers + XGBoost、Claude
Orchestration: LangChain 4-Agent
Notifications: LINE API
```
→ 読者が再現・参考にできる解像度まで落とす

### 「なぜそれを選んだか」の動機が必ず付く
- ベクトル検索 → 「前回ハッカソン優秀賞で使用した技術を全部調べた」
- 機械学習×LLM → 「救急外来では曖昧な主訴の患者さんが多い」（ドメイン知識から）
- ウェアラブル → 「研修医の仕事の流れを確認。目視で呼吸確認→脈触知→問診。これをAIで行う方法を考えた」

### 数値で裏付け
- 「救急搬送の48.5%が軽症（消防庁統計）」
- 「大病院の7割が赤字」

### ユーザー操作フローをステップ番号で説明
```
1: 入力欄に主訴を入力
2: 機械学習で主訴分類
3: グラフ表示
4: 追加問診表示
5: トリアージ結果
```

## 実践 Tips のフォーマット（開発セクション）

番号付きリスト＋太字ラベルで整理：

```
①ブラウザとサーバーのログの保存を指示
②モデルの段階的エスカレーション（Sonnet → Opus → Codex）
③ローカルサーバーはnohupで立ち上げる
④Plan modeの使用
```

各Tipは「問題 → 解決策 → 理由」の3段構成。
「なぜこれが必要か」のエピソードが付く（「非エンジニアの大きな挫折がエラー」など）。
