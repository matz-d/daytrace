# 伝えきれなかったこと（記事参照用）

発表・審査時に時間の都合で伝えきれなかった内容や、
作っている途中でしか体験できないことを記録する。

---

## 1. skill-miner は「正直な AI」を目指した

表面的には「反復パターンを提案する機能」に見えるが、
内側で目指したのは「**提案数の多さより、提案の妥当性と説明可能性を優先する AI**」だった。

これは開発中に問題として発見した：
> 75 session packets 中 63 が 1 クラスタに吸い込まれ、「根拠の弱い候補」が正式提案に混入した

この経験から、以下を設計に盛り込んだ：
- **提案件数を固定しない**（0 件でも正常系）
- **強い候補がないときは正直にそう言う**（「今回は有力候補なし」）
- **見送った提案も decision log に残す**（隠さない、消さない）

「やらないことを明示する AI」の設計は、ハッカソンの発表では伝わりにくい。

---

## 2. 自律性は「止まらないこと」

「一度命じたら、あとは任せろ」というテーマに対して、
DayTrace の自律性解釈は少しずれているかもしれない。

DayTrace の自律性 =「**最後まで進めること**」

- ソースが欠けても止まらない（Graceful Degrade）
- 設定が足りなくても、あるものだけで形を作る
- 人に確認するのは「影響の大きい決定だけ」

これは「質問しない AI」ではなく「**壊れない AI**」の設計思想。
毎日使えるツールには、これが必要だと思っている。

---

## 3. 「使うほど育つ」ループの設計

DayTrace を使い続けると何が起きるか：

```
毎日 /daytrace-session
  → 反復パターンが蓄積
  → skill-miner が候補を提案
  → skill-applier で CLAUDE.md / skill / hook / agent に適用
  → 次のセッションから自動化される
  → また別の反復が蓄積...
```

これは「日報を作るツール」ではなく、「**開発環境が自分仕様に育っていくループ**」。

提出時には 5 つのスキルとして分離したが、
本当の価値は「この循環が始まった後」にある。

## 4. ローカル完結の思想的背景

「OAuth や SaaS に頼らない」は単なる技術的制約ではない。

- 「設定不要」にするためには、ユーザーに何も持ってもらう必要がない
- 「インストールしたら即使える」が実現できる
- プライバシー：ローカルログが外に出ない

一方、これがトレードオフを生む：
- Google Calendar や Slack のデータは読めない
- 自動投稿はできない
- 「全部自動化」ではなく「材料を整えて人が判断する」スタイル

このトレードオフは意図的な選択であり、「責務の分離」として設計した。

## 5. 開発プロセスそのものが DayTrace で記録されていた

皮肉（？）なことに、DayTrace の開発プロセス自体が DayTrace によって記録されている。

`~/.daytrace/output/` には開発期間中の日報が蓄積している。
- skill-miner の品質問題を発見した日の記録
- marketplace に公開した日の記録
- クラスタリング改善が入った日の記録

「自分が作っているツールで、自分の作業が記録されている」——
これはデモでは見せにくいが、実際の体験として一番伝えたいことかもしれない。
